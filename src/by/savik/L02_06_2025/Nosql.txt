NoSQL — это класс баз данных, 
предназначенных для работы с большими объемами данных,
которые не требуют строгой реляционной структуры,
характерной для SQL-баз. 
 - SQL: данные хранятся в таблицах с фиксированной схемой
- NoSQL: гибкая схема или отсутствие схемы, 
  данные хранятся в различных форматах

 SQL (Structured Query Language)
- **Зачем нужны?** 
Для масштабируемости, 
обработки больших объемов данных,
работы с неструктурированными или
полуструктурированными данными.

 Основные типы NoSQL баз данных
1. **Ключ-значение (Key-Value)**
 - Пример: Redis, DynamoDB.
 - Используются для кэширования, сессий, простых запросов.
 - Данные хранятся как пары ключ-значение (похоже на словарь).
 - Пример: `user:123 -> {name: "Иван", age: 20}`.

2. **Документоориентированные (Document Stores)**:
 - Пример: MongoDB, CouchDB.
- Данные хранятся в виде документов (JSON, BSON),
  каждый документ — это автономная единица.
 - Подходят для приложений с динамическими данными,
   например, каталоги товаров.
- Пример: `{ "id": 123,
 "name": "Иван", 
"orders": [
	   { "item": "книга", "price": 500 },
           { "item": "книга2", "price": 600 }
	 ] 
	}`.
3. **Колоночные (Column-Family)**:
 - Пример: Cassandra, HBase.
 - Данные организованы в столбцы, а не строки
 - Используются для больших данных, аналитики, временных рядов
- Пример: данные о транзакциях,
  где столбцы — это метки времени или категории.
4. **Графовые (Graph Databases)**:
 - Пример: Neo4j, ArangoDB.
- Хранят данные в виде узлов и связей, 
  идеальны для сетей (социальные сети, рекомендации).
- Пример: `Иван -> [друг] -> Петр`.


 

### Когда использовать NoSQL?
- Большие объемы данных (Big Data).
- Неструктурированные данные (логи, посты в соцсетях).
- Высокая нагрузка
- Гибкие или часто меняющиеся структуры данных.
- Примеры: соцсети (Facebook — Cassandra), 
  интернет-магазины (Amazon — DynamoDB).

### Когда неиспользовать NoSQL?
Финансовые системы (банки, бухгалтерия).
Приложения с четко определенной структурой данных.
Сложные аналитические запросы с множеством связей.


CAP-теорема (или теорема Брюера)
— это фундаментальный принцип в 
  области распределенных систем
 который помогает понять компромиссы 
 при выборе баз данных.
Она утверждает, что распределенная система
 может гарантировать только **две из трех** 
 характеристик: 
**Consistency (Консистентность)**, 
**Availability (Доступность)** и 
**Partition Tolerance (Устойчивость к разделению)**. 
1. **Consistency (Консистентность)**:
Во всех узлах в один момент времени данные одинаковые
2**Availability (Доступность)**:
В любой момент времени при обращении к БД, мы хотим
что бы она всегда отвечала нам.
3. *Partition Tolerance (Устойчивость к разделению)**.
система продолжает работать даже если есть потеря связи 
м/у ее частями.

CP -> предпочитают согласованность данных, жертвую доступностью
HBase, MongoDB,Neo4j, ClickHouse, Обычно Redis
AP -> предпочитают доступность жертвуя согласованностью
Elasticsearch, Cassandra, CouchBase, DynamoDB, 
Redis, MongoDB

и аналитическая система, предназначенная 
для полнотекстового поиска, 
анализа логов и 
обработки неструктурированных данных.
